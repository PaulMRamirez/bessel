Independent cross-check of the Bessel build against SPEC.md

Generated 2026-06-14. Synthesis of independent verification agents (build gates, per-phase audits 0 through 5, budgets and security, and the suite contract check). Branch autonomous-run.

This report is an independent verification pass over the Bessel codebase, cross-checking the implemented build against the requirements in SPEC.md and the binding conventions in CLAUDE.md. Every status below is grounded in concrete file evidence or a captured command exit code. Findings that could not be substantiated by a file path or a command result were dropped during synthesis. Where an agent reported a command failing, this report records that failure rather than smoothing it over.

Overall verdict: PARTIAL. Phase 0 is complete and well tested. Phase 1 and Phase 2 are partial, with their named completion-condition tests passing but several build-list items missing. Phase 3 is substantially incomplete (only meta-kernel resolution is real). Phases 4 and 5 are not started. Seven of eight build gates pass on real exit codes; the single failing gate is pnpm release:dry (exit 1) because no changesets exist yet. Budgets and security checks all pass and no secrets or bulk kernels are committed.

7. Open risks and recommended follow-ups

  • Release pipeline is the one failing gate. pnpm release:dry exits 1 because no changesets exist and there is no electron-builder config or release.yml workflow. Recommended: add a changeset (or changeset add --empty for the current state), wire a true version+publish dry-run, add electron-builder configs, and generate a CHANGELOG. This is the Phase 5 release item and must clear before GA.
  • Phase scope drift. The most recent commit is Phase 3a, yet several Phase 1 and Phase 2 UI build-list items (object browser, settings panel, keyboard shortcuts, geometric readouts, multi-select, timeline annotations, screen capture/recording) were never built. Recommended: reconcile the phase ledger so a phase is not marked done while its build list has unimplemented, untested features; backfill these before advancing.
  • @bessel/scene has no test files. FOV cones and footprints are exercised only through e2e pixel assertions. The missing GLTF spacecraft, axis triads, direction vectors, and track-along-trajectory camera mode are entirely untested because unimplemented. Recommended: add scene unit tests alongside the Phase 1 scene completion.
  • Capability flags without backing implementations. fileDialogs: true and pythonBridge: true in pal-electron assert capabilities that no code provides. Recommended: gate capability flags on real implementations so the Capabilities contract does not over-promise to the UI.
  • Desktop preload exposes no kernel/filesystem IPC. preload.ts surfaces only {platform, versions}, so the Electron shell cannot yet drive native kernel loading through a typed IPC surface as Phase 1 item 6 specifies.
  • Untested implemented code paths. The OPFS kernel cache (Phase 2) and pal-capacitor importKernelZip (Phase 3) are implemented but lack dedicated unit tests; they rely on e2e or are unexercised. Recommended: add focused unit tests so these are covered independently of the browser e2e run.
  • Missing Electron e2e project. playwright.config.ts has only a chromium project, so the Phase 3 Electron meta-kernel/DSK render condition cannot ever pass without adding an Electron project and a DSK fixture.
  • Phases 4 and 5 are greenfield. Telemetry adapters, plugin registry, collaboration, WebXR, error boundaries, diagnostics, and the production release pipeline are all unstarted. These represent the bulk of remaining GA work and their completion-condition tests do not yet exist.

End of report. All statuses trace to a cited file path, line reference, or captured command exit code; unsubstantiated claims were dropped during synthesis.
