# Phase 5: Production hardening and GA

Maps to SPEC.md Section 9, Phase 5. Prerequisite: Phase 4 committed and green.
Start from a clean Git tree.

Intent: certify the program objective: a fully featured, production quality,
efficient application ready to join the NASA-AMMOS suite alongside MMGIS.
This phase hardens rather than adds features: budgets enforced
everywhere, dependency audit clean, accessibility certified, release pipeline
exercised end to end, suite integration contracts tested, and documentation
complete.

Note on signing: certificate-based signing (macOS notarization, Windows code
signing, app store submission) requires human-held secrets and accounts, so the
verifiable conditions stop at unsigned artifacts plus a pipeline that runs in
dry-run mode. The human completes signing with real certificates afterward.

## Goal body

```
/goal Implement Bessel Phase 5 as defined in SPEC.md Section 9.

Build:
1. Release pipeline: electron-builder configurations producing installable
   artifacts for Linux, macOS, and Windows (unsigned; signing hooks present but
   skippable), and a release workflow driven by changesets that versions
   packages, generates changelogs, and publishes with an npm dry-run.
2. Budgets everywhere: ensure pnpm size passes on the final bundle set, and the
   Lighthouse assertions in lighthouserc.json pass against the production
   build.
3. Hardening: error boundaries around every panel; a typed, user-facing error
   surface for kernel, catalog, and network failures; an opt-in, privacy-plain
   diagnostics report (no telemetry by default).
4. Accessibility: full keyboard operability for the object browser, timeline,
   and settings; the e2e a11y scan reports zero serious or critical violations
   across the primary views.
5. Suite contract: integration tests asserting the inbound view URL contract
   and the outbound MMGIS URL construction against fixture configs
   (docs/integrations.md).
6. Documentation: README quickstart verified by a script that executes its
   commands; CONTRIBUTING, SECURITY, and CODE_OF_CONDUCT present and linked;
   a CHANGELOG generated by changesets.

Goal is complete when ALL of the following exit 0 or hold:
- pnpm verify exits 0 (typecheck, lint, test, build:web, size)
- pnpm build:desktop succeeds and electron-builder produces unsigned artifacts
  for the host platform in dry-run mode
- pnpm audit:prod exits 0 (no high or critical production vulnerabilities)
- pnpm lhci exits 0 against the production web build
- pnpm e2e exits 0, including the a11y scan (zero serious or critical
  violations) and the suite URL contract tests
- pnpm release:dry exits 0 (changesets version and publish dry-run)

Scope: apps/desktop packaging config, a release workflow, error-surface code in
packages/ui and apps, e2e additions, documentation files, changesets.

Do not: add new visualization features in this phase. Do not delete or skip
tests. Do not weaken typecheck or budgets to pass; if a budget is genuinely
wrong, stop and raise it rather than editing .size-limit.json or
lighthouserc.json. Do not edit docs/adr/, CLAUDE.md, or AGENTS.md. Do not commit
secrets or certificates. Do not use em dashes.

Stop after 40 turns and report remaining work if not complete.
```

## Natural split point

Phase 5a: hardening, accessibility, and suite contract tests. Phase 5b: the
release pipeline, packaging artifacts, and documentation verification. Commit
between them.
