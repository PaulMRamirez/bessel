Bessel: Implementation & Gap-Analysis Report

SPICE-aware 3D mission-visualization (Apache-2.0; PWA + iOS/Capacitor + desktop/Electron; TypeScript strict, Three.js, CSPICE-WASM). Generated 2026-06-17.
Executive summary. Bessel has implemented its analysis-engine layer as 14 numerically validated headless core packages, with all Phase B and C domains (access, lighting, comms/RF, conjunction, coverage, attitude, mission/Lambert, OEM interop, 2D ground track) now surfaced in an Analysis workbench UI and proven end-to-end by Playwright e2e on the Cassini sample. Every physics quantity is pinned to an independent reference (SGP4-VER sub-meter, Vallado Lambert 7-5, EPSG:3857, NAIF occult/gfoclt, analytic Pc/eclipse/footprint), so the algorithms are demonstrably correct on fixtures. The honest positioning: Bessel reaches parity on the SHAPE of each STK domain but exercises a thin, often-hardcoded slice. The real remaining gap versus STK is the heavy numerical and estimation tier (HPOP force-model propagation, orbit determination, an optimizing Astrogator-class Mission Control Sequence, large-catalog SSA screening), plus the absent automation SDK and the cross-cutting F3 cancellable-job worker pool that gates every heavy sweep.

Scorecard: how Bessel compares to each reference

Ansys STK (Systems Tool Kit) Premium Space / STK Pro

Useful parity on shape, trails on the numerical/estimation tier

Bessel reaches demonstrable end-to-end parity on the SHAPE of each STK analytical domain (the algorithms exist and are correct on a fixture), and ten domains are surfaced into an Analysis workbench proven by e2e, but it exercises only a thin, often-hardcoded slice (single LOS constraint, fixed DSN params, illustrative Pc covariance, Walker 24/3/1, umbra-only eclipse, single-rev Lambert, equirectangular-only map). STK remains far ahead on fidelity, scale, and breadth in every domain that requires a numerical force-model integrator (HPOP, Astrogator finite burns/optimizers, OD), large-catalog screening (CAT all-vs-all), or a general automation/SDK surface. Bessel's structural edges STK cannot match: open license, zero-install offline PWA from one codebase, SPICE-native data model, and a reproducible headless runner once built. The single biggest cross-cutting blocker is the absent F3 cancellable-job worker pool. Three validated cores (propagator, sensors, terrain) and one (map-projection) are library-only or shadowed by parallel shell code despite the status table implying otherwise.

NASA GMAT (R2026a, Apache-2.0) as a potential backend

Seam-only oracle/backend, never a browser runtime

Per the binding spec decision, GMAT is NOT a runtime dependency and cannot be one for the PWA or iOS: GMAT's native stack (wxWidgets, f2c Fortran, CSPICE, Xerces-C, PCRE2, SOFA, SWIG, pthreads) has no clean Emscripten build, so a browser-side GMAT is impractical and would defeat Bessel's offline-PWA core value. GMAT is relevant in three roles only: (R1, deferred) an optional desktop/server high-fidelity ComputeProvider behind the PAL seam, gated on three unmet conditions; (R2) an offline committed-fixture validation oracle that keeps CI GMAT-free; (R3) an algorithm/test-case reference. Its value is concentrated and deep exactly where Bessel is thinnest: numerical propagation/HPOP, full Astrogator-class mission/maneuver design, and orbit determination. GMAT does NOT supply most of Bessel's STK-parity surface (no coverage grids, comms/RF, large-catalog conjunction, 2D mapping, terrain, reporting workbench). The shared CSPICE/SPK substrate makes R2 low-cost and high-leverage today; R1 stays seam-only until the decision gate is met, with OD as the single most likely future justification.

Cosmographia (NASA/JPL SPICE-enhanced 3D visualization)

Exceeds: adds a validated analysis layer + modern web app

Bessel is at or beyond Cosmographia across the core viewer surface and is a net exceed on platform, UX, analysis, and interoperability. The SPICE/geometry engine (11/11), geometry taxonomy (6/7 Done), measurement, and modern UI/UX are at or past parity, and Bessel adds entire capability classes Cosmographia lacks: a numerically validated analysis workbench (11 domains with e2e), tri-target delivery (web/PWA + iOS + desktop vs desktop-only), shareable URL state, MMGIS deep links, CZML export, light/dark theming, and loud located errors instead of the silent jump-to-Sun defect. The residual viz gaps are content and surfacing, not missing capability: no bundled photographic NASA textures and no auto-download manager, no real CK/SCLK fixture, no model reflections, and two operational parity gaps (a headless/CLI/batch entry point and live telemetry wiring).

GMAT integration decision

Seam-only, native-first, committed-fixture oracle (binding per spec §8)

GMAT is NOT a runtime dependency for Bessel and cannot be one for the PWA or iOS targets. A browser-side GMAT is impractical: GMAT's native stack (wxWidgets, f2c Fortran, CSPICE, Xerces-C, PCRE2, SOFA, SWIG, pthreads) has no clean Emscripten build, and bundling it would defeat Bessel's offline-PWA core value (it also bundles its own CSPICE, duplicating CSPICE-WASM). GMAT is therefore relevant in exactly three roles:

  • R1 (deferred runtime backend). An optional desktop/server-only high-fidelity ComputeProvider behind the PAL seam, used for HPOP propagation, Astrogator-class mission design, and OD. Held in reserve behind a three-condition decision gate: (1) a concrete fidelity need, (2) a verified production-grade headless API, (3) a verified SPK/OEM ephemeris round-trip back into Bessel's pipeline. Conditions (2) and (3) are explicitly UNVERIFIED today.
  • R2 (validation oracle, adopt now). Offline, committed-static-fixture reference trajectories and targeting/maneuver products that keep CI GMAT-free. Low-cost and high-leverage given the shared CSPICE/SPK substrate. Set tolerances empirically against the fixtures; do NOT assume the refuted 0.05 mm/s agreement figure.
  • R3 (algorithm reference). Documented MCS / differential-corrector / optimizer / OD algorithms inform native implementations.

Recommended posture: build the ComputeProvider seam now with only the native provider; invest in R2 GMAT fixtures for propagation and mission/maneuver design; hold R1 in reserve, with orbit determination the single most likely future justification for activating it. GMAT does NOT close the broad STK gap (no coverage grids, comms/RF, large-catalog SSA, 2D mapping, terrain, or reporting workbench).
